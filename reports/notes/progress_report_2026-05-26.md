## 1. 2주 전(05-13) 시점 — 어디까지 와 있었나

05-13 보고는 "조사·협의 단계"였음.

- 기존 앱/서버 CSV 업로드 구조를 코드 기준으로 확인했음
- 베데스다 회신을 분석해 필요한 변경사항을 정리했음 (`userID` → `studyId`/`subjectId`)
- 산출물(회신 메일 초안, 분석 보고서, 샘플 데이터)만 정리한 상태였음

즉 이 시점엔 **코드는 아직 손대지 않았고**, 베데스다 회신·확인을 요청하기 직전 상태였음.

## 2. 05-13 이후 경과 (타임라인)

1. **베데스다에 회신 메일 발송** — 전송 필드(`csvfile, studyId, subjectId, battery, timestamp`) 확인, 테스트용 Study/Subject ID·신규 서버 주소 요청
2. **답변으로 명세 합의** — 엔드포인트 추가 확약, `studyId`/`subjectId` 필수, 별도 인증 없이 먼저 진행
3. **QR 매칭 구조 분석** — 작년 앱(`pbcr_source`)이 QR로 Study/Subject를 앱에 매칭하는 방식임을 확인
4. **앱 코드 1차 수정 완료** — 전송 규격을 베데스다 형식으로 전환 (이번 보고 핵심)
5. **담당자가 오늘 오후 테스트 준비** 통보 → 우리도 테스트 준비 상태로 정렬

## 3. 이번에 수정한 코드 (5개 파일)

QR 스캔은 후순위로 두고, **먼저 베데스다 서버가 데이터를 정상 수신하는지 검증**할 수 있는 최소 변경을 완료했음.

| 파일 | 변경 내용 |
|---|---|
| `DeviceInfo.kt` | 식별자를 `userID` → `studyId`/`subjectId`로 전환, 테스트용 값 자리 마련 |
| `ServerConnection.kt` | 전송 주소를 신규 베데스다 서버로, 전송 항목을 합의 형식으로 교체 |
| `AcceptService.kt` | 30분 주기 업로드 호출부를 새 식별자 기준으로 교체 |
| `LoginActivity.kt` | 로그인 화면이 새 구조로 바로 진입하도록 변경 |
| `SensorActivity.kt` | 진입 시 Study/Subject 식별자로 초기화 |

### 전송 내용 변경

```
[변경 전]  구 강원대 서버 (HTTP)
  csvfile, userID, battery, timestamp

[변경 후]  베데스다 서버 (HTTPS)
  csvfile, studyId, subjectId, battery, timestamp
```

기존 단일 `userID` 구조에서 `studyId`/`subjectId` 구조로 바꿨음.
지금은 테스트용 값으로 검증하고, 이후 QR 스캔으로 자동 입력되도록 확장할 예정임.

## 4. 오늘 오후 테스트 준비 상태

코드는 준비 완료. 베데스다로부터 받는 즉시 **두 가지만 반영하면 바로 테스트 가능**함.

1. 신규 서버 주소(엔드포인트)
2. 테스트용 Study/Subject ID

### 테스트 절차

1. 받은 주소로 단독 전송 검증 (샘플 CSV 사용)
2. 값 반영 후 디버그 빌드 → 30분 주기 종단간 전송
3. 전송 성공 로그 확인 + 베데스다 측 적재 여부 확인(담당자 협조)

## 5. 앞으로 할 것

1. **오늘**: 서버 주소·테스트 ID 수신 → 단독 검증 → 앱 종단간 테스트 → 적재 확인
2. **다음 단계**: QR 스캔 기능 추가 — 카메라로 QR을 읽어 Study/Subject를 자동 입력, 테스트용 하드코딩 제거
3. **후속**: 인증 방식 등은 업로드 검증이 안정화된 뒤 논의
